The session data design

An agent runs in a sandbox; everything it does flows out as one ordered event stream, atrium keeps a permanent copy, and the UI is rebuilt from that copy. This is the end-state — what we're building toward.

What it has to achieve

Five goals

Durable foreverSurvives Centaur's retention — a finished session is fully replayable years later.
Full fidelityConversation, diffs, generated files, and the raw model-level rollout — nothing silently dropped.
One capture pathFiles and logs ride the same event stream as everything else. No second pipeline to keep in sync.
Drift-proofFrames stored opaque, so a harness changing its format never corrupts the archive.
Joinable recordTranscript + human layer (steers, suggestions, seat history) stitched by session + a monotonic event id.

The flow, end to end

From keystroke to screen

1

Agent works in a sandbox codex / claude · container

It writes messages, runs tools, edits files, generates files, and — separately — appends to its own rich rollout log on disk.

2

Two things leave the sandbox

Everything the agent narrates goes out as a live event stream. A small sidecar captures what the stream can't carry — generated file bytes and the rollout — uploads them, and emits a reference back into the same stream.

stream: messages · tools · diffs · usage sidecar: file blobs · rollout segments → object store
Why a sidecar: a binary written by a shell command (an image, a build output) never appears in the stream as bytes — only the command does. The sidecar is the only way to capture real work-product.
3

Centaur records the stream event log · Postgres

Every frame is appended with a monotonic event_id. This is the live source of truth — but it can be redacted on a retention timer, which is exactly why we don't stop here.

4

Atrium mirrors everything the key move · session_events

A tailer reads the stream and copies every frame, verbatim and opaque, into atrium's own table — kept forever, de-duplicated by event id. The mirror, not Centaur, becomes the durable source of truth.

all frames → session_events (jsonb, indefinite)
Because the sidecar's blobs are referenced inside frames, mirroring the stream automatically captures the artifacts and rollout too — one path, not three.
5

It lands in three durable homes

The mirror fans out by data type: the transcript stays in session_events, the large blobs live in the object store, and the human/collaboration layer is recorded in atrium's existing append-only event log.

transcriptblobscollab events
6

The client rebuilds the view web · mobile

The UI folds the frames back into a transcript and merges the collab layer. Live extras with no store — the env chip, running cost — are synthesized from the frames themselves.

Source of truth

Three kinds of data, three homes

There is no single source of truth — and that's deliberate. Each kind of data lives where it belongs.

Conversation

session_events · Postgres

The transcript — messages, tool calls, diffs, reasoning, usage. The mirror is its permanent home.

Work-product & forensics

object store · content-addressed

Generated file bytes + the rich model-level rollout, as immutable blobs, referenced from the transcript.

Collaboration

events · Postgres (append-only)

Steers, suggestions, seat handoffs, threads — the human layer, joined to the transcript by session + event id.

The formats

What the data looks like

Four shapes carry the whole design. The diff is verbatim from a live gpt-5.5 session.

1 · A frame — the unit of the transcriptcaptured live One row in the stream / mirror. A file edit carries its full diff — so the Changes view is free, no extra capture.
{
  "event_id": 50231,                  // monotonic, orders + dedupes
  "type": "item.completed",
  "item": {
    "type": "fileChange",
    "changes": [{
      "kind": { "type": "update" },
      "path": "calc.py",
      "diff": "@@ -2 +2,5 @@\n     return a + b\n+\n+def subtract(a, b):\n+    return a - b\n"
    }]
  }
}
2 · The mirror row — how we keep it forever Every frame kept verbatim (raw and canonical), deduped per session — drift-proof. Replay reads the self-contained item.completed snapshots; the streaming .delta frames are live-only, kept but ignored on replay.
session_events (
  session_id        uuid,
  centaur_event_id  bigint,        // the frame's event_id
  event_kind        text,
  frame             jsonb,         // the WHOLE frame, verbatim
  UNIQUE (session_id, centaur_event_id)
)
3 · Ref-frames — how files & rollout enter the stream The sidecar uploads the bytes to the object store and drops a lightweight reference into the same stream — so the mirror captures it like any other frame.
{ "type": "artifact.captured",
  "path": "/tmp/chart.png", "kind": "image/png",
  "sha256": "9f2c…", "size": 8421, "blobRef": "s3://…/9f2c…" }

{ "type": "rollout.segment",
  "seq": 7, "blobRef": "s3://…/rollout/exe_…/0007.jsonl" }
4 · A collab event — the human layer Steers, suggestions, seat changes live in atrium's append-only event log, anchored to the transcript by the same event id.
{ "type": "session.suggestion_added",
  "session_id": "…", "anchor_event_id": 50231,
  "by": "mara", "text": "try the smaller image first",
  "disposition": null }                // driver decides; kept either way
The whole design in one line: one ordered stream of frames (with blob references) → mirrored verbatim into session_events → fanned into transcript / blobs / collab → rebuilt by the client.
The one upstream change: the sandbox sidecar (steps 2 & 3). Everything else — the mirror, the diffs, the steers, the UI — needs nothing new from Centaur.
